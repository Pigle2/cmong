# 비기능 요구사항

## 1. 성능 요구사항

### 1.1 API 응답시간

| 엔드포인트 유형 | p50 | p95 | p99 | 비고 |
|----------------|:---:|:---:|:---:|------|
| **일반 조회** (GET) | 100ms | 300ms | 500ms | 서비스 목록, 프로필 등 |
| **상세 조회** (GET) | 150ms | 400ms | 700ms | 서비스 상세, 주문 상세 등 |

| **에러 로그** | 90일 | Sentry | 에러 추적 + 스택 트레이스 |

### 7.4 에스컬레이션 정책

```
Level 1 (자동 감지 후 5분)
    │
    ├── 알림: Slack #alerts 채널 + PagerDuty
    ├── 담당: 온콜 엔지니어
    └── 대응: 15분 이내 초기 대응
         │
         ▼ (30분 이내 미해결)
Level 2 (에스컬레이션)
    │
    ├── 알림: Slack + PagerDuty + SMS
    ├── 담당: 시니어 엔지니어 + 팀 리드
    └── 대응: 즉시 참여
         │
         ▼ (1시간 이내 미해결)
Level 3 (전사 대응)
    │
    ├── 알림: 경영진 포함 전체 알림
    ├── 담당: CTO + 전 개발팀
    └── 대응: 전사 War Room 가동
```

## 9. 장애 대응 절차

### 9.1 장애 등급 분류

| 등급 | 정의 | 영향 범위 | 응답 시간 | 복구 목표 |
|:----:|------|-----------|:---------:|:---------:|
| **P1 (Critical)** | 전체 서비스 중단, 결제 불가, 데이터 유실 위험 | 전체 사용자 | 5분 | 1시간 |
| **P2 (Major)** | 주요 기능 장애 (검색 불가, 채팅 불가 등) | 다수 사용자 | 15분 | 4시간 |
| **P3 (Minor)** | 부분 기능 오류, 성능 저하 | 일부 사용자 | 30분 | 24시간 |
| **P4 (Low)** | UI 오류, 사소한 버그 | 극소수 | 1시간 | 다음 배포 |

### 9.2 장애 대응 프로세스

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  감지     │───▶│  분석     │───▶│  복구     │───▶│  검증     │───▶│  후속     │
│ (Detect) │    │(Analyze) │    │(Recover) │    │(Verify)  │    │(Follow)  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
```

**1단계: 감지 (Detect)**
| 항목 | 내용 |
|------|------|
| 자동 감지 | Datadog 알림, CloudWatch 경보, Sentry 에러 알림 |
| 수동 감지 | 고객 문의, CS팀 보고, 내부 모니터링 |
| 장애 선언 | 온콜 엔지니어가 영향도 평가 후 장애 등급 선언 |

**2단계: 분석 (Analyze)**
| 항목 | 내용 |
|------|------|
| 영향 범위 파악 | 영향 받는 서비스, 사용자 수, 데이터 영향 확인 |
| 근본 원인 분석 | 로그 분석, 메트릭 확인, 최근 변경사항 확인 |
| 커뮤니케이션 | Slack 장애 채널에 상황 공유, 이해관계자 알림 |

**3단계: 복구 (Recover)**
| 항목 | 내용 |
|------|------|
| 즉시 조치 | 롤백, 스케일 아웃, 장애 격리, 트래픽 전환 |
| 임시 조치 | Workaround 적용, 기능 비활성화 |
| 영구 수정 | 근본 원인 해결 코드 배포 |

**4단계: 검증 (Verify)**
| 항목 | 내용 |
|------|------|
| 서비스 정상 확인 | 헬스체크, 스모크 테스트, 외부 서비스 연동 확인 |
| 모니터링 강화 | 관련 지표 임시 임계값 하향 조정 |
| 장애 종료 선언 | 정상 복구 확인 후 장애 종료 선언 |

**5단계: 후속 (Follow-up)**
| 항목 | 내용 |
|------|------|
| Postmortem 작성 | 장애 발생 후 48시간 이내 |
| 재발 방지 대책 | 조치 항목(Action Item) 도출 및 담당자 배정 |
| 프로세스 개선 | 장애 대응 절차 보완, 모니터링 강화 |

### 9.3 온콜 체계

| 항목 | 정책 |
|------|------|
| **온콜 로테이션** | 주간 교대 (월요일 10:00 교대) |
| **Primary 온콜** | 1명 (모든 알림 수신, 1차 대응) |
| **Secondary 온콜** | 1명 (Primary 백업, P1 장애 시 동시 참여) |
| **업무 시간 내** | 전 개발팀 대응 가능 |
| **업무 시간 외** | 온콜 엔지니어 PagerDuty 알림 |
| **온콜 보상** | 온콜 수당 + 장애 대응 시 추가 보상 |
