## 4. 실시간 통신 아키텍처

### 4.1 채팅 시스템

```
┌──────────┐     WebSocket     ┌──────────────┐
│ Client A │◀──────────────────│              │
└──────────┘                   │   Socket.IO  │
                               │   Server     │
┌──────────┐     WebSocket     │              │
│ Client B │◀──────────────────│   (Adapter:  │
└──────────┘                   │    Redis)    │
                               └──────┬───────┘
                                      │
                                      ▼
                               ┌──────────────┐
                               │    Redis     │
                               │  (Pub/Sub)   │
                               └──────────────┘
```

- Socket.IO Redis Adapter를 사용하여 다중 서버 간 메시지 동기화
- 메시지 저장은 PostgreSQL, 최근 메시지 캐시는 Redis
- 오프라인 사용자에게는 푸시 알림 발송

### 4.2 알림 시스템

```
이벤트 발생
    │
    ▼
┌──────────────┐
│ 알림 서비스   │
│ (Producer)   │
└──────┬───────┘
       │
       ▼
┌──────────────┐     ┌──────────────┐
│ BullMQ       │────▶│ 알림 워커     │
│ (알림 큐)    │     │              │
└──────────────┘     │ ├ 인앱 알림   │
                     │ ├ 푸시 (FCM)  │
                     │ ├ 이메일 (SES)│
                     │ ├ SMS (다날)  │
                     │ └ 카카오알림톡 │
                     └──────────────┘
```

## 5. 보안 아키텍처

### 5.1 인증/인가 흐름

```
Client                API Server              Auth Service
  │                       │                        │
  │  POST /auth/login     │                        │
  │──────────────────────▶│                        │
  │                       │  Verify credentials    │
  │                       │───────────────────────▶│
  │                       │◀───────────────────────│
  │                       │                        │
  │  Set HttpOnly Cookie  │                        │
  │  (Access + Refresh)   │                        │
  │◀──────────────────────│                        │
  │                       │                        │
  │  GET /v1/orders       │                        │
  │  Cookie: token=...    │                        │
  │──────────────────────▶│                        │
  │                       │  Verify JWT            │
  │                       │  Check permissions     │
  │  200 OK + data        │                        │
  │◀──────────────────────│                        │
```

### 5.2 보안 정책

| 항목 | 정책 |
|------|------|
| **HTTPS** | 전 구간 TLS 1.3 적용 |
| **CORS** | 허용 도메인 화이트리스트 |
| **CSRF** | SameSite Cookie + CSRF Token |
| **XSS** | CSP 헤더, 입력값 이스케이프 |
| **SQL Injection** | Prisma ORM 바인딩 변수 사용 |
| **Rate Limiting** | IP + 사용자 기반 제한 |
| **파일 업로드** | MIME 타입 검증, 크기 제한, 바이러스 스캔 |
| **비밀번호** | bcrypt (cost factor 12) |
| **민감 데이터** | AES-256 암호화 (계좌번호, 카드번호 등) |
| **감사 로그** | 관리자 모든 활동 기록 |

## 7. 확장성 전략

### 7.1 수평 확장

| 계층 | 전략 |
|------|------|
| **API 서버** | ECS Auto Scaling (CPU/메모리 기반) |
| **DB** | Read Replica, 테이블 파티셔닝 |
| **캐시** | Redis Cluster (샤딩) |
| **검색** | Elasticsearch 클러스터 노드 추가 |
| **파일** | S3 (자동 확장) + CloudFront |

### 7.2 성능 최적화

| 영역 | 전략 |
|------|------|
| **DB 쿼리** | 쿼리 최적화, 적절한 인덱스, N+1 방지 |
| **캐싱** | Redis 캐시 (서비스 목록, 카테고리, 인기 서비스) |
| **CDN** | 정적 자원/이미지 CDN 캐싱 |
| **이미지** | WebP 변환, 리사이즈, lazy loading |
| **API** | 페이징, 필드 선택(sparse fieldsets), 압축(gzip) |
| **프론트** | 코드 스플리팅, SSR/SSG, 번들 최적화 |

### 7.3 가용성/장애 대응

| 전략 | 설명 |
|------|------|
| **Multi-AZ** | RDS, ElastiCache, ECS를 2개 이상 AZ에 배포 |
| **Health Check** | ALB + ECS 헬스 체크, 자동 복구 |
| **PG Failover** | 결제 PG사 장애 시 대체 PG사 자동 전환 |
| **Circuit Breaker** | 외부 서비스 장애 시 Fallback 처리 |
| **Backup** | RDS 자동 스냅샷 (일 1회), S3 버전닝 |
| **DR** | 일간 전체 백업, 크로스 리전 복제 (선택) |

## 8. 모니터링/로깅

| 지표 | 도구 | 알림 기준 |
|------|------|-----------|
| API 응답 시간 | Datadog | p95 > 1초 |

| Redis 메모리 | CloudWatch | > 70% |
| 큐 지연 | BullMQ Dashboard | > 100건 대기 |
| 에러 발생 | Sentry | Critical 에러 즉시 |

### 8.2 로그 구조
