## 1. 테스트 피라미드

### 1.1 테스트 계층 및 비율

```
                    ┌───────┐
                    │  E2E  │   5%
                    │       │
                ┌───┴───────┴───┐
                │  Integration  │   25%
                │               │
            ┌───┴───────────────┴───┐
            │     Unit Tests        │   60%
            │                       │
        ┌───┴───────────────────────┴───┐
        │      Static Analysis          │   10%
```

| 테스트 유형 | 비율 | 커버리지 목표 | 실행 환경 | 실행 시간 목표 |

| **services** | 서비스 등록 검증, 패키지 가격 계산 | > 85% | P1 |

| **search** | 검색 쿼리 생성, 필터 빌드, 정렬 로직 | > 80% | P1 |

### 2.4 핵심 비즈니스 로직 테스트 케이스

#### 결제 금액 계산 (BR-PAY)

| # | 테스트 케이스 | 입력 | 기대 결과 |
|---|-------------|------|-----------|
| 1 | 기본 결제 | 패키지 100,000원 | 결제금액 103,500원 (수수료 3.5%) |
| 2 | 쿠폰 적용 | 100,000원 + 5,000원 할인 쿠폰 | 결제금액 98,325원 |
| 3 | 포인트 사용 | 100,000원 + 5,000P 사용 | 결제금액 98,325원 |
| 4 | 포인트 50% 초과 | 100,000원 + 60,000P 사용 | 에러: 최대 50% 사용 가능 |
| 5 | 최소 결제 금액 미달 | 4,000원 | 에러: 최소 5,000원 (BR-PAY-03) |
| 6 | 추가 옵션 포함 | 패키지 100,000원 + 옵션 30,000원 | 결제금액 134,550원 |

#### 정산 금액 계산 (BR-PAY)

| # | 테스트 케이스 | 입력 | 기대 결과 |
|---|-------------|------|-----------|
| 1 | 개인 판매자 정산 | 서비스금액 100,000원 | 77,360원 (수수료 20% + 원천징수 3.3%) |
| 2 | 사업자 판매자 정산 | 서비스금액 100,000원 | 80,000원 (수수료 20%) |
| 3 | 빠른 정산 | 서비스금액 100,000원, 빠른 정산 | 수익 80,000원 - 빠른정산 수수료 1.5% |
| 4 | 빠른 정산 등급 제한 | 일반 등급 판매자 빠른 정산 | 에러: 프로 등급 이상만 가능 (BR-PAY-04) |
| 5 | 최소 정산 금액 미달 | 정산 금액 9,000원 | 에러: 최소 10,000원 |
| 6 | 월 정산 한도 초과 | 개인 월 정산 1,000만원 초과 | 에러: 한도 초과 (BR-PAY-06) |
| 7 | 부분 환불 후 재계산 | 50% 부분 환불 | 수수료 재계산 (BR-PAY-07) |

#### 주문 상태 전이 (BR-ORD)

| # | 테스트 케이스 | 현재 상태 | 동작 | 기대 상태 |
|---|-------------|-----------|------|-----------|
| 1 | 결제 완료 | - | 결제 성공 | `PAID` |
| 2 | 주문 수락 | `PAID` | 판매자 수락 | `ACCEPTED` |
| 3 | 주문 거절 | `PAID` | 판매자 거절 | `REJECTED` → 전액 환불 |
| 4 | 작업 시작 | `ACCEPTED` | 판매자 작업 시작 | `IN_PROGRESS` |
| 5 | 납품 | `IN_PROGRESS` | 판매자 납품 | `DELIVERED` |
| 6 | 수정 요청 | `DELIVERED` | 구매자 수정 요청 | `REVISION_REQUESTED` |
| 7 | 구매 확정 | `DELIVERED` | 구매자 확정 | `COMPLETED` |
| 8 | 자동 확정 | `DELIVERED` (5일 경과) | 배치 실행 | `COMPLETED` (BR-ORD-03) |
| 9 | 24시간 미수락 자동 취소 | `PAID` (24시간 경과) | 배치 실행 | `CANCELLED` (BR-ORD-01) |
| 10 | 잘못된 상태 전이 | `COMPLETED` | 취소 시도 | 에러: 허용되지 않는 상태 전이 |
| 11 | 분쟁 신청 | `DELIVERED` | 이의 신청 | `DISPUTED` |

### 3.2 API 엔드포인트 테스트

#### 인증 API

| 엔드포인트 | 테스트 항목 |
|-----------|------------|
| `POST /v1/auth/register` | 정상 가입, 중복 이메일, 유효성 검증 실패, 소셜 계정 연동 |
| `POST /v1/auth/login` | 정상 로그인, 잘못된 비밀번호, 비활성 계정, 정지 계정 |
| `POST /v1/auth/refresh` | 정상 갱신, 만료된 토큰, 유효하지 않은 토큰 |
| `POST /v1/auth/logout` | 정상 로그아웃, 토큰 블랙리스트 등록 확인 |

#### 서비스 API

| 엔드포인트 | 테스트 항목 |
|-----------|------------|
| `POST /v1/services` | 서비스 등록, 필수 필드 누락, 카테고리 검증, 이미지 연결 |
| `GET /v1/services` | 목록 조회, 페이징, 필터링, 정렬 |
| `GET /v1/services/:id` | 상세 조회, 존재하지 않는 ID, 비공개 서비스 접근 |
| `PUT /v1/services/:id` | 수정, 다른 사용자 서비스 수정 시도, 재심사 필요 항목 (BR-SVC-07) |

#### 주문 API

| 엔드포인트 | 테스트 항목 |
|-----------|------------|
| `POST /v1/orders` | 주문 생성, 결제 처리, 에스크로 보관, 동시 주문 제한 (BR-ORD-07) |
| `PUT /v1/orders/:id/accept` | 주문 수락, 권한 검증, 이미 처리된 주문 |
| `PUT /v1/orders/:id/deliver` | 납품 처리, 파일 첨부 확인 |
| `PUT /v1/orders/:id/confirm` | 구매 확정, 정산 큐 등록 확인, 포인트 적립 확인 |
| `POST /v1/orders/:id/dispute` | 분쟁 신청, 5일 이내 확인, 증빙 자료 첨부 |

#### 정산 API

| 엔드포인트 | 테스트 항목 |
|-----------|------------|
| `GET /v1/settlements` | 정산 내역 조회, 기간 필터, 상태 필터 |
| `POST /v1/settlements/withdraw` | 출금 요청, 최소 금액 검증, 계좌 검증 |
| `GET /v1/admin/settlements` | 관리자 정산 현황, 일괄 정산 처리 |

### 4.2 주요 사용자 시나리오

#### 시나리오 1: 서비스 구매 전체 플로우

```
1. 구매자 로그인
2. 서비스 검색 ("로고 디자인")
3. 서비스 상세 조회
4. 패키지 선택 (Deluxe)
5. 주문서 작성 (요구사항 입력)
6. 쿠폰 적용
7. 결제 (테스트 PG)
8. 주문 완료 확인
9. 판매자 주문 수락
10. 판매자 작업 시작
11. 판매자 납품
12. 구매자 검수
13. 구매자 구매 확정
14. 리뷰 작성
15. 포인트 적립 확인
16. 판매자 정산 내역 확인
```

#### 시나리오 2: 환불 플로우

```
1. 구매자 결제 완료
2. 판매자 수락 전 구매자 취소
3. 전액 환불 처리 확인
4. 쿠폰 복원 확인 (CP-06)
5. 환불 상태 확인
```

#### 시나리오 3: 분쟁 처리 플로우

```
1. 주문 납품 완료 상태
2. 구매자 이의 신청
3. 판매자 소명 작성
4. 관리자 분쟁 검토
5. 결정 (부분 환불)
6. 양측 알림 확인
7. 정산 반영 확인
```

#### 시나리오 4: 판매자 서비스 등록 플로우

```
1. 판매자 로그인
2. 서비스 등록 페이지 진입
3. 기본 정보 입력 (제목, 카테고리, 설명)
4. 이미지 업로드
5. 패키지 설정 (Standard/Deluxe/Premium)
6. 추가 옵션 설정
7. FAQ 작성
8. 서비스 등록 요청
9. 관리자 심사 대기 상태 확인
10. 관리자 승인
11. 서비스 공개 확인
```

#### 시나리오 5: 채팅 + 견적 플로우

```
1. 구매자 견적 요청 작성
2. 판매자 견적서 발송
3. 채팅으로 상세 논의
4. 견적서 기반 주문 생성
5. 결제 완료 확인
```

## 5. 성능 테스트

### 5.1 도구

| 항목 | 기술 | 비고 |
|------|------|------|
| **부하 테스트** | k6 (Grafana) | 스크립트 기반, CI 연동 |
| **모니터링** | Datadog + Grafana | 실시간 지표 확인 |
| **환경** | Staging (Production 동일 사양) | |

### 5.2 부하 테스트 시나리오

| 시나리오 | 동시 사용자 | TPS 목표 | 지속 시간 | 성공 기준 |
|---------|:----------:|:--------:|:---------:|-----------|
| **일반 부하** | 500 | 500 | 30분 | p95 < 500ms, 에러율 < 0.1% |
| **피크 부하** | 2,000 | 2,000 | 15분 | p95 < 1,000ms, 에러율 < 0.5% |
| **스파이크** | 0 → 3,000 | - | 5분 | 오토스케일링 정상 작동, 에러율 < 1% |
| **내구성** | 300 | 200 | 4시간 | 메모리 누수 없음, 응답시간 안정 |

### 5.3 주요 API 부하 테스트 항목

| API | 시나리오 | 목표 TPS | p95 목표 |
|-----|---------|:--------:|:--------:|
| `GET /v1/services` | 서비스 목록 조회 | 200 | 300ms |
| `GET /v1/search` | 검색 | 100 | 300ms |
| `POST /v1/orders` | 주문 생성 | 50 | 1,000ms |
| `POST /v1/auth/login` | 로그인 | 100 | 300ms |
| `GET /v1/services/:id` | 서비스 상세 | 300 | 400ms |
| WebSocket 연결 | 채팅 동시 접속 | - | 5,000 동시 연결 |

### 5.4 스트레스 테스트

| 항목 | 테스트 내용 | 성공 기준 |
|------|-----------|-----------|
| **DB 커넥션 풀 고갈** | 최대 커넥션 도달 시 동작 | 대기 후 처리, 에러 아님 |
| **Redis 메모리 한계** | 캐시 메모리 70% 초과 | LRU 정상 동작, 서비스 영향 없음 |
| **큐 적체** | BullMQ 1,000건 대기 | Worker 스케일 아웃, 순차 처리 |
| **디스크 사용률** | 로그 누적 시 | 로그 로테이션 정상, 서비스 영향 없음 |

## 6. 보안 테스트

### 6.1 OWASP Top 10 검증

| # | 취약점 | 테스트 항목 | 도구 |
|---|--------|-----------|------|
| A01 | **Broken Access Control** | 수평/수직 권한 상승, IDOR | 수동 + Burp Suite |
| A02 | **Cryptographic Failures** | TLS 설정, 민감 데이터 암호화 | SSL Labs, testssl.sh |
| A03 | **Injection** | SQL Injection, XSS, Command Injection | OWASP ZAP |
| A04 | **Insecure Design** | 비즈니스 로직 우회, Rate Limiting 우회 | 수동 |
| A05 | **Security Misconfiguration** | 디폴트 설정, 에러 노출, 불필요 헤더 | OWASP ZAP |
| A06 | **Vulnerable Components** | npm 의존성 취약점 | npm audit, Snyk |
| A07 | **Auth Failures** | 무차별 대입, 세션 고정, 토큰 유출 | Burp Suite |
| A08 | **Data Integrity Failures** | 서명 검증, 역직렬화 | 수동 |
| A09 | **Logging Failures** | 로그 누락, 민감 정보 로그 | 수동 |
| A10 | **SSRF** | 내부 네트워크 접근, 메타데이터 접근 | Burp Suite |

### 6.2 인증/인가 테스트

| # | 테스트 항목 | 시나리오 |
|---|-----------|---------|
| 1 | 권한 없는 리소스 접근 | 구매자가 다른 구매자의 주문 조회 시도 |
| 2 | 역할 권한 위반 | 구매자가 관리자 API 접근 시도 |
| 3 | JWT 만료 후 접근 | 만료된 Access Token으로 API 호출 |
| 4 | Refresh Token 재사용 | 이미 사용된 Refresh Token으로 갱신 시도 |
| 5 | 토큰 위조 | 시크릿 변경한 JWT로 접근 시도 |
| 6 | 관리자 2FA 우회 | 2FA 없이 관리자 기능 접근 시도 (BR-ADM-01) |
| 7 | 정지 계정 접근 | 정지 상태 계정으로 로그인/API 호출 |

### 6.3 입력값 검증

| # | 검증 항목 | 테스트 입력 |
|---|---------|-----------|
| 1 | XSS 방어 | `<script>alert('xss')</script>` (서비스 제목, 리뷰 등) |
| 2 | SQL Injection | `'; DROP TABLE users; --` |
| 3 | 파일 업로드 | 이중 확장자 (`.jpg.php`), 대용량 파일 (50MB 초과) |
| 4 | MIME 타입 위조 | 실행 파일을 이미지 MIME으로 위장 |
| 5 | 금액 조작 | 음수 금액, 소수점 이하 금액, 극대값 |
| 6 | 이메일 형식 | 비정상 이메일 형식, 극단적 길이 |
| 7 | 경로 탐색 | `../../../etc/passwd` (파일 업로드 경로) |

## 9. 회귀 테스트

### 9.1 자동화 범위

| 테스트 유형 | 자동화 범위 | 실행 조건 |
|------------|:---------:|-----------|
| **단위 테스트** | 100% | 모든 PR, 모든 push |
| **통합 테스트** | 100% | 모든 PR (develop/main 대상) |
| **E2E 테스트 (Critical Path)** | 5개 시나리오 | Staging 배포 후 자동 실행 |
| **E2E 테스트 (전체)** | 전체 시나리오 | 주 1회 (수요일 야간) |
| **성능 테스트** | 주요 API | 월 1회 (Production 동일 사양) |
| **보안 테스트** | OWASP ZAP 자동 스캔 | 월 1회 |

### 9.2 실행 주기

| 주기 | 실행 항목 |
|------|----------|
| **PR마다** | Lint + Type Check + 단위 테스트 + 통합 테스트 |
| **develop push** | 위 + Docker Build + Staging 배포 + 스모크 테스트 |
| **main push** | 위 + Production 배포 + 스모크 테스트 + E2E Critical Path |
| **주 1회** | E2E 전체 + 비주얼 리그레션 테스트 |
| **월 1회** | 성능 테스트 + 보안 스캔 + 의존성 취약점 검사 |

### 9.3 실패 시 대응 절차

| 단계 | 조건 | 동작 |
|------|------|------|
| 1 | PR 테스트 실패 | PR 머지 차단, 개발자 수정 |
| 2 | Staging 스모크 실패 | 배포 롤백, Slack 알림 |
| 3 | Production 스모크 실패 | 즉시 롤백, P2 장애 선언 |
| 4 | E2E 전체 실패 | Slack 알림, 다음 영업일 내 수정 |
| 5 | 보안 스캔 Critical | 48시간 이내 수정 |
